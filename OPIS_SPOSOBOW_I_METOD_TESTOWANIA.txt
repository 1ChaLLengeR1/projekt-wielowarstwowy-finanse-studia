 1. PRZEGLĄD STRATEGII TESTOWANIA
----------------------------------------------------------------

W projekcie zastosowano trzy uzupełniające się metody testowania:

  1. Testy jednostkowe backendu — pytest (automatyczne)
  2. Testy E2E frontendu     — Cypress (automatyczne)
  3. Manualne testy API      — Swagger UI /docs (ręczne)

Podejście to pokrywa zarówno izolowaną logikę (jednostkowe),
jak i integrację warstw (manualne) oraz zachowanie UI (E2E).

 2. TESTY JEDNOSTKOWE BACKENDU (pytest)
----------------------------------------------------------------

  Technologia: pytest 8.3.5
  Uruchomienie: pytest -s -v -m ""
                (z katalogu backend/, po aktywacji środowiska)

  2.1 WARSTWA HELPERY — nagłówki HTTP
  ------------------------------------
  Plik: tests/core/helper/test_headers.py
  Testowana funkcja: check_required_headers() z core/helper/headers.py

  Scenariusze testowe (10 testów):

    TC-H01  Brak jednego wymaganego nagłówka
            Wejście: pusty słownik headers, wymagany ["Authorization"]
            Oczekiwany wynik: status=ERROR, status_code=403,
            komunikat zawiera nazwę brakującego nagłówka.

    TC-H02  Brak kilku wymaganych nagłówków jednocześnie
            Wejście: pusty headers, wymagane ["Authorization","UserData"]
            Oczekiwany wynik: ERROR 403, obie nazwy w komunikacie.

    TC-H03  Pusta lista wymaganych nagłówków
            Wejście: pusty headers, wymagane []
            Oczekiwany wynik: SUCCESS 200, data=[].

    TC-H04  Nagłówek UserData z poprawnym JSON
            Wejście: UserData: {"id":"123","role":"admin"}
            Oczekiwany wynik: SUCCESS 200, sparsowany obiekt w data.

    TC-H05  Nagłówek UserData z niepoprawnym JSON
            Wejście: UserData: "not-valid-json"
            Oczekiwany wynik: ERROR 403, "Invalid JSON format" w komunikacie.

    TC-H06  Nagłówek X-Refresh-Token
            Wejście: X-Refresh-Token: "some-token-value"
            Oczekiwany wynik: SUCCESS 200, wartość tokenu w data.

    TC-H07  Oba nagłówki UserData i X-Refresh-Token jednocześnie
            Oczekiwany wynik: SUCCESS, obie wartości w data.

    TC-H08  Nagłówek Authorization nie trafia do data (tylko walidacja)
            Wejście: Authorization: "Bearer token123"
            Oczekiwany wynik: SUCCESS 200, data=[].

    TC-H09  Jeden z dwóch wymaganych nagłówków brakuje
            Wejście: mamy Authorization, brak UserData
            Oczekiwany wynik: ERROR 403, komunikat o UserData,
            Authorization nie wymieniony jako brakujący.

    TC-H10  Nagłówek UserData z pustym obiektem JSON {}
            Oczekiwany wynik: SUCCESS, data[0].data == {}.

 4. MANUALNE TESTOWANIE API — Swagger UI (/docs)
----------------------------------------------------------------

  Adres: http://localhost:3000/docs
  (backend uruchomiony lokalnie lub przez Docker)

  Swagger UI generowany automatycznie przez FastAPI na podstawie
  typów Pydantic i dekoratorów routerów. Umożliwia testowanie
  każdego endpointu bezpośrednio z przeglądarki.


  4.1 PROCEDURA LOGOWANIA I AUTORYZACJI W SWAGGER
  -------------------------------------------------
  Krok 1. Otwórz http://localhost:3000/docs
  Krok 2. Rozwiń sekcję "Authentication"
  Krok 3. Wywołaj POST /authentication/login
          Body: {"username": "<user>", "password": "<pass>"}
          Oczekiwany wynik: HTTP 200, access_token + refresh_token
  Krok 4. Kliknij "Authorize" (kłódka, prawy górny róg)
          Wklej: Bearer <access_token>
  Krok 5. Wszystkie kolejne endpointy wywołuj już z autoryzacją.


  4.2 SCENARIUSZE MANUALNYCH TESTÓW — MODUŁ ZADAŃ
  -------------------------------------------------
  MT-T01  POST /tasks/create
          Body: {"description": "Test zadania", "time": 30, "active": true}
          Oczekiwany wynik: HTTP 200, data.id (UUID), data.description.

  MT-T02  GET /tasks/collection?active=true
          Oczekiwany wynik: HTTP 200, lista aktywnych zadań.
          Weryfikacja: nowo utworzone zadanie z MT-T01 jest na liście.

  MT-T03  PATCH /tasks/update/{task_id}
          Path param: id z MT-T01
          Body: {"description": "Zmieniony opis", "time": 60}
          Oczekiwany wynik: HTTP 200, zaktualizowane dane.

  MT-T04  PATCH /tasks/update/active/{task_id}
          Body: {"active": false}
          Oczekiwany wynik: HTTP 200, data.active == false.
          Weryfikacja: GET /tasks/collection?active=false — zadanie widoczne.

  MT-T05  GET /tasks/statistics?start_date=2025-01-01&end_date=2025-12-31
          Oczekiwany wynik: HTTP 200, klucze total_tasks, total_time,
          average_per_week, average_time_per_week, tasks_per_day.

  MT-T06  DELETE /tasks/delete/{task_id}
          Oczekiwany wynik: HTTP 200, data.id == usuniętego zadania.
          Weryfikacja: GET collection — zadanie zniknęło z listy.

  MT-T07  Nieistniejące ID — DELETE /tasks/delete/<random-uuid>
          Oczekiwany wynik: HTTP 404, status=ERROR.
